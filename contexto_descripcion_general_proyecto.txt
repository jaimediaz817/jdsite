# 1. Arquitectura general

## Patrón arquitectónico

El proyecto sigue una **arquitectura en capas con separación de responsabilidades** típica de aplicaciones Django enterprise:

```
┌─────────────────┐
│   Frontend      │  Templates Django + JavaScript vanilla
│   (Vistas)      │  Bootstrap 4.3.1 + CSS custom
├─────────────────┤
│   Backend       │  URL dispatcher → Views → Services → Models
│   (Lógica)      │  Comandos de gestión para tareas batch
├─────────────────┤
│   Datos         │  MySQL (producción) / SQLite (desarrollo)
│   (Persistencia)│  ORM Django con migrations versionadas
├─────────────────┤
│   Externa       │  Allauth (OAuth), Resend/SES (email),
│   (Servicios)   │  GitHub API, AWS S3 (media producción)
└─────────────────┘
```

## Organización del proyecto

- **Monorepo Django** con múltiples apps: `blog`, `core`, `inquiries`, `reactions`
- **Fuente de verdad**: Archivos Markdown en `blogs_source/` (no la BD)
- **Capa de cache**: Base de datos como índice de búsqueda y metadatos
- **Procesamiento batch**: Comando de gestión `import_blogs` sincroniza fuentes → BD → HTML

## Comunicación entre capas

1. **Request** → URL Router (`urls.py`)
2. **View** → Template o JSON response
3. **Service Layer** (`services.py`) → Lógica de negocio compleja
4. **Model** → ORM Django con validaciones
5. **Management Command** (`import_blogs`) → Pipeline ETL: Markdown → HTML → BD → Archivos estáticos

## Escalabilidad

- **Horizontal**: Stateless backend (excepto media temporal), desplegado con Gunicorn + Nginx
- **Datos**: MySQL con índices optimizados, separación de estáticos/media
- **Código**: Comandos de gestión desacoplados, services layer reusable
- **Infraestructura**: Variables de entorno por ambiente, logging estructurado

## Comandos de gestión

### `import_blogs` (comando principal)
Pipeline ETL completo:
1. Lee archivos `.md` de `blogs_source/`
2. Extrae frontmatter YAML
3. Procesa bloques especiales (`:::slides`, `:::callout`, `:::popup:gallery`)
4. Convierte Markdown → HTML con `markdown.markdown()`
5. Extrae imagen de portada
6. Copia archivos estáticos a `static/blogs/<slug>/`
7. Actualiza/crea registros en BD
8. Genera tokens de aprobación para borradores

### `sanitize_images`
Normaliza nombres de archivo eliminando espacios y caracteres especiales.

### `backfill_created_at`
Migración de datos para poblar `created_at` en posts legacy.

# 5. Editor de artículos

## Capacidades

### Creación
- Formulario completo con título, descripción, contenido Markdown
- Selección de categoría y tags
- Subida de imágenes y videos
- Asignación de portada manual (`cover_filename`)
- Slug opcional (HU-045)

### Edición
- Carga de artículo existente desde `blogs_source/`
- Merge inteligente de frontmatter ( preserva campos custom )
- Preservación de autor original (no sobrescribe en ediciones)
- Historial de cambios (implícito por `source_hash`)

### Markdown
- **Sintaxis completa**: Títulos, listas, código,Links, imágenes
- **Bloques especiales**: Ver sección 4
- **Imágenes**: `![alt](filename.png)` → rutas relativas
- **Videos**: YouTube embed + widget custom

### Subida de imágenes
- **Formatos**: PNG, JPG, JPEG, GIF, WEBP, SVG, MP4, WEBM, MOV, AVI
- **Límite**: Configurable via `MAX_UPLOAD_SIZE_MB` (default 20MB)
- **Procesamiento**: 
  - Renombrado con UUID para evitar colisiones
  - Sanitización de nombres
  - Movimiento de `/media/blog_editor_temp/` a `blogs_source/<slug>/`
- **Validación**: Tipo MIME, tamaño, extensión

### SEO (en editor)
- **Meta title** y **meta description** editables
- **Palabra clave principal**
- **Keywords** adicionales
- **Tiempo de lectura** (calculado automáticamente en import)
- **Slug** optimizado para URLs

### Categorías y Tags
- **Categorías**: Obligatorio,FK a modelo Category
- **Tags**: Opcionales, M2M con modelo Tag
- **Auto-creación**: Si no existen, se crean en import

### Publicación
- **Flujo**: Borrador → Pendiente moderación → Aprobado/Rechazado
- **Publicador**: Superadmin publica automáticamente
- **Aprobación**: Token único en URL para aprobación sin login
- **Notificaciones**: Email al autor y admin en cada cambio de estado

### Borradores
- **Creación**: Por defecto, artículos nuevos son borradores
- **Identificación**: Campo `draft: true` en frontmatter + `is_published=False` en BD
- **Listado**: Dashboard muestra borradores y publicados
- **Filtrado**: Por estado de moderación

## Procesamiento post-guardado

Al guardar, se ejecuta automáticamente:
1. `save_blog_to_source()` escribe el `.md` en `blogs_source/`
2. `call_command("import_blogs")` se ejecuta
3. `import_blogs`:
   - Copia imágenes a `static/blogs/<slug>/`
   - Genera HTML
   - Actualiza BD
   - Genera tokens

## Desasociación

- **Lógica en `save()`**:
  ```python
  if self.is_active and self.blog_post:
      QRCode.objects.filter(
          blog_post=self.blog_post, 
          is_active=True
      ).exclude(pk=self.pk).update(is_active=False)
  ```

# 7. Autenticación

## Métodos implementados

### Google OAuth 2.0
- **Provider**: `allauth.socialaccount.providers.google`
- **Scope**: `profile`, `email`
- **Configuración**: Variables `GOOGLE_OAUTH_CLIENT_ID` y `GOOGLE_OAUTH_CLIENT_SECRET`
- **Flujo**: 
  1. Usuario click "Login with Google"
  2. Redirige a Google
  3. Callback con code
  4. Allauth obtiene token y perfil
  5. Auto-signup o login

### GitHub OAuth
- **Provider**: `allauth.socialaccount.providers.github`
- **Scope**: `read:user`, `user:email`
- **Configuración**: Variables `GITHUB_OAUTH_CLIENT_ID` y `GITHUB_OAUTH_CLIENT_SECRET`
- **Flujo**: Similar a Google

### Usuario y contraseña
- **Formulario**: `ACCOUNT_SIGNUP_FIELDS` con email, nombre, username, password
- **Login**: Por email (`ACCOUNT_AUTHENTICATION_METHOD = "email"`)
- **Registro**: Manual con validación de campos

### Recuperación de contraseña
- **Allauth**: Flujo nativo de recuperación por email
- **Email backend**: Configurable (consola, SMTP, AWS SES)

### Cambio de contraseña
- **Vista**: No custom, usa Allauth
- **URL**: `/account/password/change/`

### Verificación
- **Email verification**: Deshabilitada (`ACCOUNT_EMAIL_VERIFICATION = "none"`)
- **Razón**: OAuth automático, sin verificación manual

## Seguridad

- **CSRF**: Protección en todos los formularios
- **HTTPS**: En producción (`SECURE_SSL_REDIRECT = True`)
- **Secure cookies**: `CSRF_COOKIE_SECURE`, `SESSION_COOKIE_SECURE`
- **Proxy header**: `SECURE_PROXY_SSL_HEADER` para detectar HTTPS detrás de Nginx

# 15. Funcionalidades diferenciadoras

## ¿Qué hace diferente a un blog tradicional?

### 1. **Fuente de verdad en archivos, no en BD**
- Los artículos viven en `blogs_source/*.md`
- La BD es solo un índice para búsquedas rápidas
- **Demuestra**: Comprensión de arquitectura Git-friendly, versionado de contenido

### 2. **Pipeline ETL propio (`import_blogs`)**
- Conversión Markdown → HTML con bloques especiales
- Sincronización automática de archivos estáticos
- **Demuestra**: Capacidad de construir sistemas de procesamiento de contenido

### 3. **Sistema de moderación con tokens**
- Aprobación de borradores por URL sin login
- Flujo completo: Borrador → Pendiente → Aprobado
- **Demuestra**: Diseño de workflows de aprobación

### 4. **QR codes para artículos físicos**
- Generación automática con logo
- Desvinculación inteligente
- **Demuestra**: Pensamiento omnicanal (digital + físico)

### 5. **Editor con bloques especiales**
- No es un Markdown básico
- Componentes enriquecidos: slides, videos, galerías
- **Demuestra**: Capacidad de extender formatos de contenido

### 6. **Sistema de autoría y notificaciones**
- Preservación de autor original
- Notificaciones granulares (admin vs autor)
- Configuración centralizada (`BlogEmailConfig`)
- **Demuestra**: Diseño de sistemas multi-tenant con notificaciones

### 7. **Arquitectura multi-ambiente**
- Desarrollo: SQLite, DEBUG=True
- Producción: MySQL, HTTPS, logging detallado
- **Demuestra**: Conocimiento de despliegues profesionales

4. **Producción ready**: HTTPS, CSRF, logging, variables de entorno, Nginx + Gunicorn

# 17. Resumen ejecutivo

Este proyecto demuestra la capacidad de construir sistemas web completos, desde la base de datos hasta el despliegue en producción. No es un blog Django genérico: es una plataforma de publicación con arquitectura multi-capa, pipelines de procesamiento de contenido custom, sistema de moderación con aprobación por tokens, integración de códigos QR para distribución física, y notificaciones granulares por email.

El desarrollador demuestra dominio de Django a nivel intermedio-avanzado: servicios desacoplados, comandos de gestión para ETL, ORM optimizado con índices estratégicos, y configuración multi-ambiente sensible. La elección de mantener los archivos Markdown como fuente de verdad, en lugar de depender exclusivamente de la base de datos, refleja comprensión de arquitecturas Git-friendly y versionado de contenido.

En frontend, implementa un sistema de bloques especiales en Markdown que va más allá del formato básico, demostrando capacidad de extender parsers y diseñar lenguajes de contenido custom. El sistema de QR codes para artículos físicos muestra pensamiento omnicanal y atención a casos de uso reales de distribución de contenido.

En infraestructura, el despliegue en DigitalOcean con Nginx + Gunicorn, HTTPS con Certbot, y configuración por variables de entorno indica experiencia en puesta en producción. La documentación de procedimientos y scripts de mantenimiento sugiere mentalidad de operaciones.

El proyecto tiene margen de crecimiento hacia arquitectura API-first con DRF, procesamiento asíncrono con Celery, y containerización con Docker, pero como está, es un sistema funcional, mantenible y listo para producción que resuelve problemas reales de publicación y moderación de contenido.

**Evaluación objetiva**: El desarrollador construyó un sistema end-to-end con separación de responsabilidades, no un blog tutorial. La complejidad está en los detalles: merge inteligente de frontmatter, desvinculación automática de QRs, flujo de aprobación con tokens, y pipelines de importación que sincronizan fuentes de archivos con bases de datos. Eso requiere comprensión sólida de Django, patrones de diseño, y operaciones en producción.
