# HU-011.15‑A Experiencia de Usuario en el Dashboard

## Descripción general (contexto)
En la vista del *dashboard* de moderación, los **autores** pueden crear, editar o publicar artículos, pero el **superadmin** no ve esos cambios en tiempo real. Actualmente debe recargar manualmente la página para actualizar los contadores y la lista de artículos. Queremos mejorar la experiencia añadiendo un botón de **Refrescar** visible solo para superadmin, siguiendo buenas prácticas de UI/UX.

> **⚠️ Nota:** Estamos en fase de definición. No se implementará ninguna funcionalidad adicional en este momento; solo se está planificando y documentando la mejora.

## Objetivo
Permitir que el superadmin actualice la información del dashboard con un solo clic, sin necesidad de usar el refresco del navegador.

## Criterios de aceptación
1. En la barra superior del dashboard aparece un botón **Refrescar** solo para usuarios con `is_superuser`.
2. El botón tiene estilo profesional (icono de sincronización, colores coherentes con la UI existente).
3. Al pulsar el botón, la página se recarga completamente.
4. No se introducen cambios funcionales en la lógica del backend.
5. Las pruebas existentes siguen pasando.

## Pasos de implementación
1. Modificar `backend/blog/templates/blog/dashboard.html` para añadir el botón y su script de recarga.
2. Añadir la lógica de visibilidad (`{% if user.is_superuser %}`) en el template.
3. Crear este documento de historia de usuario y marcarlo como **Pendiente**.
4. Ejecutar la suite de pruebas para asegurar que no se rompen funcionalidades.
5. **Pendiente avanzado**: Analizar e implementar actualizaciones en tiempo real (por ejemplo, usando Django Channels y WebSockets) para que el superadmin vea automáticamente los nuevos artículos sin necesidad de refrescar manualmente.

## Pendiente avanzado (real‑time)
**Objetivo:** Que el superadmin vea automáticamente los nuevos artículos creados por cualquier autor sin necesidad de refrescar la página.

**Posibles soluciones a evaluar:**
1. **Django Channels** + WebSockets: crear un consumer que notifique al cliente cuando se cree o actualice un `BlogPost`.
2. **Server‑Sent Events (SSE)**: una alternativa más ligera si solo se necesita enviar eventos unidireccionales.
3. **Polling AJAX** con intervalo corto: solución menos compleja pero menos eficiente.

**Criterios de aceptación (para la fase avanzada):**
* El dashboard muestra en tiempo real el incremento del contador `total` y la aparición del nuevo artículo en la tabla.
* La implementación no degrada significativamente el rendimiento del servidor ni la experiencia de usuarios normales.
* Se añaden pruebas unitarias/integración que verifiquen la emisión y recepción de eventos.

**Tareas pendientes:**
- [ ] Investigar viabilidad de Django Channels en el proyecto (dependencias, configuración de ASGI).
- [ ] Definir arquitectura de eventos (qué eventos emitir, payload, canales).
- [ ] Implementar prototipo básico y validar con pruebas manuales.
- [ ] Documentar la solución y actualizar la HU con los resultados.
