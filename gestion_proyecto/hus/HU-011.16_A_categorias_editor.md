## Descripción general (contexto)
En el editor online de artículos, el campo **Categoría** está limitado a un `<select>` que solo muestra las categorías ya existentes. Un autor que quiera crear una categoría nueva se ve forzado a solicitar a un super‑admin que la añada, lo que frena el flujo de trabajo.

## Objetivo
Permitir que los autores normales puedan **escribir libremente** el nombre de una categoría en el editor. El sistema debe:
1. **Buscar coincidencias** entre la cadena escrita y las categorías existentes (búsqueda case‑insensitive y `icontains`).
2. Si existen coincidencias, **sugerir** al autor que seleccione una de ellas.
3. Si no hay coincidencias, **crear** la nueva categoría y asociarla al artículo que se está guardando.

## Criterios de aceptación
1. En el formulario del editor se mantiene el `<select>` con las categorías existentes y se añade un **checkbox** "No encontré categoría, quiero crear una nueva". Al marcar el checkbox se habilita un **campo de texto** opcional para introducir el nombre de la nueva categoría.
2. Al enviar el formulario, el backend valida lo siguiente:
   - Si se ha seleccionado una categoría del `<select>`, se usa esa categoría.
   - Si el checkbox está marcado y se ha introducido un nombre en el campo de texto, el backend comprueba si ya existe una categoría con ese nombre (búsqueda case‑insensitive). Si existe, se asocia esa categoría al post; si no, se crea la nueva categoría y se asocia.
3. En caso de que el nombre introducido ya exista, el backend **no crea** una nueva categoría y simplemente asocia la existente al artículo.
4. Sólo usuarios con permiso de **author** pueden crear categorías nuevas; los super‑admin siguen con el comportamiento actual.
5. Se añaden pruebas unitarias que cubran los distintos flujos: selección de categoría existente, creación de nueva categoría y detección de categoría duplicada.

## Pasos de implementación
1. **Frontend** (`blog_editor.html` y `blog_editor.js`):
   - Reemplazar el `<select id="category">` por `<input list="category-options" id="category" autocomplete="off">` y crear un `<datalist id="category-options">` poblado con las categorías actuales.
   - Añadir lógica JS que, al enviar el formulario, haga `POST /api/categories/validate/` con `{name: <valor>}`.
   - Gestionar la respuesta: mostrar modal de sugerencias o toast de creación.
2. **Backend**:
   - Crear `CategoryValidateView` (APIView) que reciba `name`, busque coincidencias y devuelva JSON según los criterios.
   - Extender el serializer de `Post` para aceptar `category_name` además de `category_id`.
   - Modificar la vista de creación/edición de post para usar la lógica anterior: si `category_id` no está presente, validar/crear vía la API.
   - Añadir permiso `IsAuthorOrReadOnly` que permita crear categorías sólo a usuarios con rol author.
3. **Modelo** (`Category`): asegurar que el campo `name` sea único (case‑insensitive) mediante `UniqueConstraint`.
4. **Pruebas**:
   - Test de la API de validación (coincidencias y creación).
   - Test de flujo completo: crear post con categoría nueva y con categoría sugerida.
5. **Documentación**:
   - Actualizar la HU en `gestion_proyecto/hus/` y añadir notas en la documentación del proyecto.
