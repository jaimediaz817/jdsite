## Descripción general (contexto)
En el editor online de artículos, el campo **Categoría** es un `<select>` que solo muestra categorías ya existentes. Un autor que quiera crear una categoría nueva se ve forzado a solicitar a un super-admin que la añada, lo que frena el flujo de trabajo.

## Objetivo
Permitir que los autores puedan **escribir libremente** el nombre de una categoría cuando no encuentren la deseada en el select. El sistema debe:
1. Mantener el `<select>` con categorías existentes como opción principal.
2. Ofrecer un **checkbox** "No encontré la categoría que busco" que al marcarse, oculte el select y habilite un **campo de texto** para escribir la categoría libremente.
3. Al guardar el artículo, el backend (vía `import_blogs`) ya maneja automáticamente la validación y creación de categorías nuevas con búsqueda case-insensitive.

### Flujo de datos actual (ya funcional)

```
Editor (JS)
  ↓ collectFormData() → { category: "Django" } (string)
  ↓ POST /blog/api/save-blog/
  ↓
services.py → save_blog_to_source()
  ↓ new_fm["category"] = "Django"  (guarda en frontmatter YAML)
  ↓ call_command("import_blogs")
  ↓
import_blogs → get_or_create_category()
  ↓ 1. Busca por name__iexact (case-insensitive)
  ↓ 2. Si no existe, busca por name__icontains palabra por palabra
  ↓ 3. Si aún no existe, crea la categoría con get_or_create()
  ↓
  ✅ Categoría creada y asociada al BlogPost
```

### Hallazgo clave
**La creación automática de categorías YA EXISTE en `import_blogs.py` línea 1037-1045:**
```python
if not category_obj:
    category_obj, _ = Category.objects.get_or_create(
        name__iexact=category_input,
        defaults={
            "name": category_input,
            "is_active": True,
            "slug": slugify(category_input),
        },
    )
```
Esto significa que **cualquier string** que el usuario escriba como categoría será procesado correctamente por import_blogs, que creará la categoría si no existe.

## Criterios de aceptación
1. En el formulario del editor se mantiene el `<select>` con las categorías existentes.
2. Debajo del select se añade un **checkbox** "No encontré la categoría que busco".
3. Al marcar el checkbox se **oculta el select** y se **habilita un campo de texto** para escribir el nombre de la nueva categoría libremente.
4. Al desmarcar el checkbox se **muestra el select** y se **oculta el campo de texto** (y se limpia su valor).
5. La validación del formulario acepta **evidentemente** que o bien el select tiene un valor O el input de texto tiene un valor (nunca ambos vacíos).
6. Al enviar el formulario, el valor de la categoría (sea del select o del input de texto) se envía como string en el JSON.
7. El backend `import_blogs.get_or_create_category()` procesa el string: si la categoría ya existe la asocia; si no existe, la crea automáticamente.
8. La categoría nueva aparecerá en el select la próxima vez que se abra el editor.
9. No se rompe el flujo de edición de artículos existentes.

### Compatibilidad con import_blogs
- `get_or_create_category()` busca primero por `name__iexact` (exacto case-insensitive).
- Si no lo encuentra, busca por `name__icontains` palabra por palabra.
- Si aún no lo crea, usa `get_or_create()` para crear la categoría.
- Si el frontmatter no tiene categoría, usa "Sin Categoria" como fallback.

### Modelo Category
```python
class Category(models.Model):
    name = models.CharField(max_length=100, unique=True)
    slug = models.SlugField(max_length=120, unique=True, db_index=True)
    description = models.TextField(blank=True, null=True)
    is_active = models.BooleanField(default=True)
    
    def save(self, *args, **kwargs):
        if not self.slug:
            self.slug = slugify(self.name)
        super().save(*args, **kwargs)
```

### El campo category en el frontmatter
Se guarda como string simple: `category: "Django"` (con comillas en YAML).
import_blogs lee este string y ejecuta la lógica de get_or_create.
