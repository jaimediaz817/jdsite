## Descripción general (contexto)
Cada vez que se ejecuta el comando `import_blogs`, se generan **carpetas duplicadas** en `backend/static/blogs/` con nombres como:
- `2026-04-26_mejoras_ui_ux_blog_historico-copy/`
- `2026-04-26_mejoras_ui_ux_blog_historico-copy-3/`
- `test_blog-copy-1/`
- `test_blog-copy-10-1/`
- etc.

Actualmente hay **~140+ carpetas** en `backend/static/blogs/`, muchas de ellas duplicados. Además, `backend/static/blogs/` **NO está excluido del .gitignore**, lo que causa que aparezcan en `git status` como archivos sin trackear.

## Objetivo
1. **Diagnosticar y corregir** la causa raíz de la duplicación de carpetas.
2. **Limpiar** las carpetas duplicadas y huérfanas existentes.
3. **Excluir** `backend/static/blogs/` del `.gitignore` para evitar que aparezcan cambios pendientes.
4. **Crear una vista en el dashboard** (solo superadmin) que muestre los recursos (imágenes, videos) de cada artículo desde `backend/static/blogs/`.
5. **Futuro**: ampliar para permitir ver también los recursos de `backend/blogs_source/<slug>/`.

## 🔍 Diagnóstico de la causa raíz

### Flujo actual que genera duplicados

```
save_blog_to_source() (services.py)
  → folder_name = f"{today}_{slug}"  (ej: "2026-04-06_mi-nuevo-articulo")
  → target_dir = source_dir / folder_name
  → shutil.move(files from temp → target_dir)
  → call_command("import_blogs")
     → process_single_blog(blog_dir)
        → slug = slugify(blog_dir.name)  (ej: "2026-04-06-mi-nuevo-articulo")
        → blog_static_dir = self.STATIC_TARGET / slug
        → self.copy_blog_images(blog_dir, blog_static_dir)  ← SIEMPRE copia
```

### Problema identificado

1. **`copy_blog_images()` siempre copia imágenes**, incluso si el archivo ya existe en `backend/static/blogs/<slug>/`.
2. **La lógica de `replacement` no es idempotente**: cada ejecución crea carpetas nuevas (`-copy`, `-copy-2`, etc.) o sobreescribe existentes.
3. **`cleanup_removed_blogs()` solo limpia la BD (BlogPost)**, NO las carpetas estáticas en `backend/static/blogs/`.
4. **`.gitignore` NO excluye `backend/static/blogs/`**, generando cambios pendientes en `git status`.
5. **No hay verificación** de si los archivos ya existen en destino antes de copiar.

### Causa específica de los nombres `-copy-N`

En `process_single_blog()`:
```python
slug = slugify(blog_dir.name)
```
Si el directorio tiene nombre `2026-04-26_mejoras_ui_ux_blog_historico`, slug es `2026-04-26-mejoras-ui-ux-blog-historico`.

Pero si se ejecuta `import_blogs` con carpetas con sufijos (por ejemplo `-copy`, `-1`, `-2`), cada una genera un slug diferente → `2026-04-26-mejoras-ui-ux-blog-historico-copy`, `-copy-3`, etc.

## Notas técnicas

### Flujo de datos actual
```
blogs_source/slug/blog.md → import_blogs → coaches into static/blogs/slug/
blogs_source/slug/image.png → import_blogs → copies to static/blogs/slug/image.png
```

### Causa de los sufijos `-copy-N`
Al ejecutar `import_blogs` después de que el editor ha movido archivos a `blogs_source/` con nombres duplicados:
1. `services.py` → `save_blog_to_source()` → genera slug con prefijo de fecha
2. `import_blogs` → procesa cada carpeta en `blogs_source/`
3. Si hay carpetas con sufijos (pruebas manuales o errores), se generan slugs diferentes
4. `copy_blog_images` crea carpetas en `static/blogs/` con el slug correspondiente
