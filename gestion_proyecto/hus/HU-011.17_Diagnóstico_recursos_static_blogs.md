## Descripción general (contexto)
Cada vez que se ejecuta el comando `import_blogs`, se generan **carpetas duplicadas** en `backend/static/blogs/` con nombres como:
- `2026-04-26_mejoras_ui_ux_blog_historico-copy/`
- `2026-04-26_mejoras_ui_ux_blog_historico-copy-3/`
- `test_blog-copy-1/`
- `test_blog-copy-10-1/`
- etc.

Actualmente hay **~140+ carpetas** en `backend/static/blogs/`, muchas de ellas duplicados. Además, `backend/static/blogs/` **NO está excluido del .gitignore**, lo que causa que aparezcan en `git status` como archivos sin trackear.

### Concepto clave: Archivos huérfanos
Un **archivo huérfano** es cualquier archivo o carpeta en `backend/static/blogs/` que **NO tiene un BlogPost asociado** en la base de datos. Pueden existir por:
- Artículos creados en fase de pruebas y luego eliminados sin limpiar archivos.
- Ejecuciones múltiples de `import_blogs` que generan carpetas con sufijos.
- Errores en `save_blog_to_source()` que dejan archivos temporales.
- Migraciones incompletas entre carpetas de `blogs_source/`.
- Todo aquello que implique usar javascript, hacerlo en los archivos dashboard.js y lo mismo para los estilos css, procuremos NO implementar javascript ni CSS desde la plantilla HTML.

## Objetivo
1. **Diagnosticar y corregir** la causa raíz de la duplicación de carpetas.
2. **Limpiar** las carpetas duplicadas y huérfanas existentes.
3. **Excluir** `backend/static/blogs/` del `.gitignore` para evitar que aparezcan cambios pendientes.
4. **Ampliar el dashboard existente** (solo superadmin) con una sección que muestre los recursos (imágenes, videos) de cada artículo.
5. **Detectar y mostrar archivos huérfanos** con indicador visual para que el admin pueda identificarlos.
6. **Permitir eliminar archivos huérfanos** desde el dashboard mediante selección con checkbox + botón de eliminación.
7. **Visualizar también los recursos de `backend/blogs_source/<slug>/`** (carpeta fuente de los artículos) en el dashboard de superadmin, con la misma interfaz y patrones visuales.

### Concepto de "Compilación" (proceso completo de import_blogs)

La "compilación" es **todo el pipeline ejecutado por `import_blogs`** tras bambalinas. No es solo copiar archivos: es un proceso completo que transforma la fuente en contenido servible al público.

**La fuente de verdad** de cada artículo es `backend/blogs_source/<slug>/`. Contiene:
- `blog.md` (Markdown + frontmatter YAML)
- Imágenes y videos originales (PNG, JPG, MP4, etc.)

**Pipeline completo de compilación (`import_blogs`):**

```
1. 📖 Lectura: Lee blogs_source/<folder>/blog.md
2. 📋 Parseo: Extrae frontmatter YAML (título, categoría, tags, SEO, etc.)
3. 📝 Conversión: Convierte Markdown → HTML (con extensiones: extra, codehilite, tables, fenced_code)
4. 🔧 Procesamiento de bloques especiales:
   - :::slides → slider con navegación
   - :::callout:info/warning/tip → cajas informativas
   - :::pullquote → cita destacada
   - :::codefile → bloque de código con header
   - :::popup:gallery → galería popup
5. 🖼️ Procesamiento de imágenes: Resuelve rutas, copia a static/blogs/<slug>/
6. 🎬 Procesamiento de videos: Convierte img tags a <video> HTML5
7. 🎠 Auto-carruseles: Detecta imágenes consecutivas y crea carruseles Swiper
8. 📊 SEO: Trunca meta_title (59) y meta_description (154)
9. 🏷️ Tags: Crea/obtiene Tag objects en BD
10. 📂 Categoría: Crea/obtiene Category object en BD (get_or_create)
11. 👤 Autor: Resuelve autor por email, author_id o preserva existente
12. 💾 Guardado: Crea/actualiza BlogPost en BD con HTML compilado
13. 🔄 Limpieza: Elimina blogs que ya no existen en filesystem
14. 📋 Resumen: Muestra estadísticas de importación
```

**Resultado de la compilación:**
```
blogs_source/           →  Código fuente (Markdown + imágenes originales)
  blog.md               →  Archivo fuente con frontmatter YAML
  imagen.png            →  Recurso original

import_blogs            →  Compilador completo (pipeline de 14 pasos)

static/blogs/<slug>/    →  Build output (assets copiados para servir)
  imagen.png            →  Recurso compilado/copiado
  blog.html             →  HTML renderizado (en BD, no en disco)

BlogPost (BD)           →  Registro compilado (title, content_html, category, tags, etc.)
```

**En el dashboard de superadmin se mostrará un "mapa de compilación" con 3 columnas:**
1. **📂 Fuente (blogs_source/)**: qué carpetas existen y sus archivos
2. **⚙️ Compilado (static/blogs/)**: qué carpetas existen y sus archivos
3. **🔗 Estado de asociación (BD)**: qué BlogPosts existen y a qué carpetas apuntan

**Indicadores visuales:**
- 🟢 **Verde**: Fuente → Compilado → BD (todo asociado y sincronizado)
- 🟡 **Amarillo**: Fuente existe pero no tiene Compilado (falta import_blogs)
- 🔴 **Rojo**: Compilado existe pero no tiene BD (carpeta huérfana)
- 🔵 **Azul**: Fuente existe pero no tiene BlogPost asociado (skip o error)

## 🔍 Diagnóstico de la causa raíz

### Flujo actual que genera duplicados

```
save_blog_to_source() (services.py)
  → folder_name = f"{today}_{slug}"  (ej: "2026-04-06_mi-nuevo-articulo")
  → target_dir = source_dir / folder_name
  → shutil.move(files from temp → target_dir)
  → call_command("import_blogs")
     → process_single_blog(blog_dir)
        → slug = slugify(blog_dir.name)  (ej: "2026-04-06-mi-nuevo-articulo")
        → blog_static_dir = self.STATIC_TARGET / slug
        → self.copy_blog_images(blog_dir, blog_static_dir)  ← SIEMPRE copia
```

### Problema identificado (CAUSA RAÍZ)

**El Deber Ser:** Al editar un artículo existente, el sistema debe **reescribir/actualizar** el contenido del blog.md y sus recursos en la misma carpeta. Nunca crear copias.

**Lo que realmente sucede:**

1. **`save_blog_to_source()`** en `services.py` genera un `folder_name = f"{today}_{slug}"` para el nuevo artículo. En **edición** (`is_edit=True`), debería encontrar la carpeta existente y sobreescribirla, pero **no siempre lo hace correctamente**.
2. **`import_blogs`** se ejecuta después de cada guardado y procesa **TODAS** las carpetas en `blogs_source/`. Si hay carpetas con sufijos (`-copy`, `-1`, `-2`), cada una genera un slug diferente → se crea un BlogPost nuevo en BD y se copian imágenes a `static/blogs/<slug>/`.
3. **`copy_blog_images()` siempre copia imágenes**, incluso si el archivo ya existe en `backend/static/blogs/<slug>/`. No verifica si el archivo ya está en destino.
4. **`cleanup_removed_blogs()` solo limpia la BD (BlogPost)**, NO las carpetas estáticas en `backend/static/blogs/`.
5. **`.gitignore` NO excluye `backend/static/blogs/`**, generando cambios pendientes en `git status`.

**En resumen:** El sistema está diseñado para que `blogs_source/` sea la fuente de verdad, y `import_blogs` la sincronice con BD + static. El problema es que se crean carpetas duplicadas en `blogs_source/` (por el editor o por pruebas) y `import_blogs` las procesa todas como si fueran artículos nuevos.
