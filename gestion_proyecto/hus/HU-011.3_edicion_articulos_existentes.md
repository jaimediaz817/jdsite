# HU-011.3: Edición de artículos existentes preservando formato

## Contexto
Al editar artículos existentes desde el editor, algunos campos del frontmatter (cabecera) no se guardan correctamente, provocando pérdida de metadatos importantes como la imagen de portada, meta descripción, etc. Es crítico poder actualizar contenido sin romper la estructura de formato ya establecida.

## Diagnóstico actual

### Flujo actual (solo creación)
1. El editor (`blog_editor.html`) envía: title, description, category, tags, meta_title, meta_description, palabra_clave_principal, keywords, tiempo_lectura, content_md, files[]
2. `save_blog_to_source()` en `services.py` siempre crea una **nueva carpeta** `YYYY-MM-DD_slug/` y un nuevo `blog.md`
3. No existe endpoint para cargar un artículo existente en el editor
4. No existe lógica para detectar si se está editando vs creando

### Campos del frontmatter generados
```yaml
title, description, date, draft, image, author, author_email, author_provider, 
category, tags, meta_title, meta_description, keywords, tiempo_lectura, palabra_clave_principal
```

### Problemas identificados
1. **No hay campo `slug`** en el editor para identificar edición
2. **`save_blog_to_source`** siempre crea carpetas nuevas
3. **No hay vista para cargar** un artículo existente en el editor
4. **No hay endpoint PUT/PATCH** para actualizar

## Objetivo
- Permitir editar artículos existentes sin perjudicar el formato actual.
- Garantizar que todos los campos del frontmatter se guarden al actualizar.
- Mantener la compatibilidad con el procesador Markdown actual.

## Criterios de aceptación
1. Al abrir un artículo existente en el editor, se cargan todos los campos del frontmatter.
2. Al guardar cambios, se preserva el formato del frontmatter sin alteraciones no deseadas.
3. Los campos que no se editan permanecen con sus valores originales.
4. La imagen de portada y otros campos específicos se guardan correctamente.
5. No se generan conflictos con el formato markdown existente.
