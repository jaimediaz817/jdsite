# HU-011.4 — Consistencia Total Markdown ↔ Editor Online

> **Origen:** Tras múltiples sesiones de uso, se detectó que el editor online y la creación manual por markdown divergen en el frontmatter y en el manejo del autor. Esto genera pérdida de información (campos custom, autor original) y confusión sobre el concepto de "borrador" vs "edición publicada".
>
> **Objetivo:** Garantizar que un artículo pueda ser creado en markdown y editado en el editor online (o viceversa) sin perder NINGÚN dato, con semántica clara de borrador vs publicación.

---

## 🟡 REGLAS DE ORO DE ESTA HU

1. **El frontmatter del .md es la fuente de verdad** — el editor solo añade/edita campos, **nunca borra campos desconocidos**.
2. **El `author` solo se sobreescribe si el editor guarda como "nuevo dueño"** — por defecto se preserva.
3. **Separación semántica clara**: existe "Borrador" (no se muestra al público) y "Publicar/Actualizar" (se muestra).
4. **El `cover_filename` explícito del editor tiene prioridad** sobre la primera imagen subida.

---

## 🔍 DIAGNÓSTICO DE PROBLEMAS ACTUALES

### Problema 1: Regeneración destructiva del frontmatter
**Archivo:** `backend/blog/services.py` (líneas 290-308)

```python
frontmatter = f"""---
title: "{title}"
description: "{description}"
...
tiempo_lectura: {tiempo_lectura}
palabra_clave_principal: "{palabra_clave_principal}"
---
"""
```

Si el `.md` original tiene campos extra (`# comentario`, `reimport 2026-06-01...`, `draft_notes`, etc.), se **pierden al guardar desde el editor**.

### Problema 2: Autor se sobreescribe siempre
**Archivo:** `backend/blog/services.py` (línea 188-189)

```python
author = user if user.is_authenticated else None
```

Aunque el `.md` ya tuviera `author_email: "jaime@x.com"`, al guardar el editor lo reescribe con el email del usuario que está editando.

### Problema 3: Sin distinción Borrador vs Actualizar
**Archivo:** `backend/blog/static/blog/js/blog_editor.js`

El botón siempre dice "🟡 Guardar Borrador". No hay opción "Actualizar artículo" para una edición normal.

### Problema 4: `cover_filename` ignorado
**Archivo:** `backend/blog/services.py` (líneas 261-269)

```python
if filetype == "image" and not image_filename:
    image_filename = filename
```

Solo toma la primera imagen. Si el usuario marcó una como portada, se ignora.

## 📦 FASES DE IMPLEMENTACIÓN (15 min cada una)

### Fase 1: Preservar campos desconocidos del frontmatter
- **Acción:** Refactor `save_blog_to_source` para hacer merge inteligente.
- Lógica nueva:
  1. Si `is_edit`, leer el frontmatter actual del `.md`.
  2. Crear un dict con los campos actuales.
  3. Sobrescribir SOLO los campos que el editor conoce (`title`, `description`, `cover_filename`, etc.).
  4. Mantener cualquier campo extra tal cual (preservar `reimport 2026-06-01`, `draft_notes`, `custom_field`, etc.).
  5. Reescribir el archivo con el merge.
- **Validación:** Cargar `2026-04-24_por-que-las-integraciones-zoho-fallan/blog.md`, editar y guardar, verificar que `# reimport 2026-06-01` siga en el frontmatter.

### Fase 2: Preservar `author_email` si ya existe
- **Acción:** Modificar `save_blog_to_source` para no sobreescribir el `author_email` ni el `author_name` del frontmatter si ya existen.
- Lógica nueva:
  1. Si `is_edit` y el frontmatter actual tiene `author_email`, mantenerlo (no usar `user.email`).
  2. Si el artículo es **nuevo** (`is_edit=False`), sí usar el `user.email` del editor.
  3. Loggear con WARNING cuando se preserve el autor original.
- **Validación:** Editar un blog del usuario A desde la cuenta de A, verificar que el frontmatter no cambie de autor.

### Fase 3: Implementar concepto "Publicar" vs "Borrador" en el editor
- **Acción:** Refactor del botón de guardar.
- Lógica nueva:
  - Para **admin** (`is_superuser`): el botón principal dice "💾 Actualizar artículo" (en edición) o "🟢 Publicar artículo" (en creación).
  - Para **no-admin**: el botón principal dice "📝 Guardar borrador" (en creación) o "🔄 Actualizar borrador" (en edición).
  - Agregar un checkbox "Publicar inmediatamente" para no-admins que les permita publicar.
- **Validación:** Verificar que el draft:`true`/`false` se respete.

### Fase 4: Usar `cover_filename` explícito del editor
- **Acción:** Modificar `save_blog_to_source` línea 261.
- Lógica nueva:
  1. Si el frontend envía `cover_filename` y existe en `files_list`, usar ese como `image_filename`.
  2. Si no, mantener el fallback actual (primera imagen).
- **Validación:** Subir 3 imágenes, marcar la tercera como portada, guardar y verificar que `image:` en frontmatter sea la tercera.

### Fase 5: Sincronización bidireccional robusta
- **Acción:** Documentar y probar el flujo completo.
- Verificar:
  1. Crear artículo en markdown → abrir en editor → editar → guardar.
  2. Crear artículo en editor → verificar `.md` → editar markdown → reimportar.
  3. Editar artículo dos veces seguidas (no debe duplicar contenido).
- **Validación:** Crear 3 artículos de prueba, cada uno con un campo custom en el frontmatter, y verificar que sobreviven.
