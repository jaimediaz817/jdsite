# HU-011.85 — Configuración centralizada de envío de emails en el blog

> **Origen:** Actualmente el blog envía correos electrónicos de forma implícita en dos flujos:
> 1. **Comentarios:** Cuando un usuario comenta un post, se envía un email al admin con los datos del comentario y un enlace para moderarlo.
> 2. **Borradores:** Cuando un usuario no-admin guarda un borrador, se envía un email al admin con los datos del artículo y un enlace de aprobación rápida.
>
> Ambos flujos usan `send_mail()` directamente en `services.py` sin ninguna configuración centralizada. Si el admin quiere dejar de recibir correos, no tiene forma de hacerlo sin modificar código. Además, el admin podría estar revisando manualmente los borradores y no necesitar la notificación por email, lo que generaría ruido innecesario.

> **Objetivo:** Implementar una **tabla paramétrica** (`BlogEmailConfig`) que actúe como interruptor maestro para el envío de emails. Desde la dashboard del superadmin se podrá activar/desactivar el envío de correos de forma global. La dashboard mostrará el estado actual del interruptor con un indicador visual animado.

---

## 🟡 REGLAS DE ORO

1. **Tabla paramétrica `BlogEmailConfig`:** Un solo registro que almacena la configuración global de envío de emails.
2. **Campo `emails_enabled` (BooleanField):** `True` = se envían correos, `False` = no se envían. Default: `True` (para no romper el comportamiento actual).
3. **Solo superadmin puede modificar:** La configuración solo es accesible y modificable por usuarios con `is_superuser=True`.
4. **Indicador visual en dashboard:** Un ícono de correo en la barra superior del dashboard que pulsa lentamente cuando está activo y permanece estático en color de alerta cuando está desactivado.
5. **Página de configuración:** Un botón en la dashboard que lleva a una página de configuración con el switch ON/OFF y una descripción del estado actual.
6. **Cero ruptura:** El comportamiento actual de envío de emails se mantiene intacto. Solo se añade una verificación previa al `send_mail()`.
7. **Extensible:** La tabla está diseñada para futuras configuraciones de email (destinatarios, frecuencia de resúmenes, etc.).

## 🔧 PLAN DE IMPLEMENTACIÓN (fases de ~15 min cada una)

### Fase 1: Modelo `BlogEmailConfig` (~10 min)
- Crear el modelo en `backend/blog/models.py`:
  ```python
  class BlogEmailConfig(models.Model):
      emails_enabled = models.BooleanField(
          default=True,
          help_text="Interruptor maestro para el envío de emails del blog"
      )
      updated_at = models.DateTimeField(auto_now=True)
      updated_by = models.ForeignKey(
          settings.AUTH_USER_MODEL,
          on_delete=models.SET_NULL,
          null=True, blank=True,
          help_text="Último usuario que modificó la configuración"
      )
      
      class Meta:
          verbose_name = "Configuración de emails del blog"
          verbose_name_plural = "Configuración de emails del blog"
      
      def __str__(self):
          return f"Emails: {'ON' if self.emails_enabled else 'OFF'}"
      
      @classmethod
      def get_config(cls):
          """Retorna la configuración singleton, creándola si no existe."""
          config, _ = cls.objects.get_or_create(pk=1)
          return config
  ```
- Crear migración: `python manage.py makemigrations blog`
